# Совместный редактор документа + ИИ-чат

Краткое описание продукта: один документ слева, чат с ИИ справа. ИИ не только отвечает в чате, но и **вносит правки в текст документа** (целиком или фрагментами). Несколько человек одновременно видят документ и чат в реальном времени; в чате видно, что набирают другие, но **чужие сообщения править нельзя**.

---

## 1. Цель

- Быстро собрать или доработать текст (Markdown или plain text — формат на выбор продукта, без принципиальной разницы для первой версии).
- Использовать ИИ как «редактора»: запрос на естественном языке → ответ в чате + применённые изменения в документе.
- Работать вместе без усложнения: без истории «кто что написал», без ролей и лишней социальной механики — **простой, предсказуемый UX**.

---

## 2. Макет экрана

| Слева | Справа |
|--------|--------|
| Редактор документа (пустой при создании) | Чат с ИИ (и коллегами в той же ленте, если так решите) |

- Кнопка **«Новый документ»**: открывает новую сессию с пустым документом и пустым (или приветственным) чатом.

---

## 3. Цикл работы с ИИ

1. Пользователь пишет в чат, что нужно (пожелание, правка, стиль, структура и т.д.).
2. Сообщение уходит в ИИ.
3. ИИ возвращает:
   - **Ответ в чате** (коротко можно пояснить, что сделал).
   - **Изменения документа**: полная замена текста или вставка/замена фрагмента — для первой версии достаточно любого одного понятного механизма (например: «новое содержимое целиком» или «патч по диапазонам» — техническая деталь реализации).
4. Правки **сразу отображаются** у всех, кто смотрит этот документ (см. раздел про реалтайм).

Авторство строк в документе **не показываем** и не храним для UI (по желанию можно логировать на бэкенде для отладки — вне скоупа «простого» интерфейса).

---

## 4. Совместная работа (реалтайм)

- **Документ**: все участники сессии видят одно и то же содержимое; правки (от людей и от ИИ) синхронизируются в реальном времени (курсоры/выделение опционально для v2; минимум — мгновенное обновление текста).
- **Чат**: все видят сообщения и обновления ленты в реальном времени.
- **Набор текста в чате**: пока коллега набирает сообщение, остальные **видят индикатор набора** (или превью черновика в read-only), но **не могут редактировать чужой черновик** — только своё поле ввода.

Итог: один общий документ + одна общая лента чата; прозрачность того, что происходит, без сложных прав доступа.

---

## 5. Принципы UX (keep it simple)

- Не усложнять: без профилей в панели, без «соавторства по абзацам», без комментариев привязанных к спанам в v1 (если не понадобится).
- Один понятный поток: документ ↔ запрос в ИИ ↔ обновление документа.
- Ошибки ИИ: показать в чате; при сбое применения текста — не ломать документ без возможности отката (откат/versioning — отдельное решение; для MVP можно начать с последней удачной версии в памяти сессии).

---

## 6. Другие документы как контекст чата (не «ссылки» в БД)

- Упоминание другого документа — это **часть контекста запроса к ИИ**, а не отдельная сущность «линк между документами» в базе и не обязательная навигационная гиперссылка в продукте.
- В UI: picker, `@` или выбор из списка — пользователь **прикрепляет к сообщению** один или несколько документов (или их фрагменты); в промпт уходит их текст как дополнительный контекст.
- В ленте чата достаточно показать, что к сообщению было «прикреплено: …» (для ясности коллегам), без модели перекрёстных ссылок в Postgres.
- Лимиты по токенам и усечение длинных вложений — на усмотрение реализации.

| Параметр | Значение |
|----------|-----------|
| **Project ID (ref)** | `jnfebpmkcejmclmoqiky` |
| **URL API** | `https://jnfebpmkcejmclmoqiky.supabase.co` |

**Как использовать Supabase в этой фиче (ориентир):**

- **Postgres** — сущности: документы, сообщения чата, привязка к «комнате»/сессии документа, опционально черновики набора (typing). Контекст из других документов живёт в **сообщении** (текст/метаданные), отдельная таблица «линков» между документами не требуется.
- **Realtime** — подписка на изменения строк таблиц и/или broadcast-каналы для низкой задержки: совместное редактирование текста, новые сообщения, индикаторы набора в чате.
- **Auth** (когда понадобится идентификация) — кто имеет доступ к документу; для MVP можно начать с общего секрета/ссылки, но проект уже под продакшен.
- **Edge Functions** (рекомендуется) — прокси к OpenAI и Anthropic: ключи провайдеров **только на сервере**, в клиент не попадают. Сюда же логирование usage (токены) для последующего точного учёта.

Переменные окружения (не коммитить реальные значения): для этой фичы в дашборде заведены отдельные имена, чтобы не путать с уже существующим `SUPABASE_URL` основного проекта — **`NEXT_PUBLIC_DOCS_SUPABASE_URL`**, **`NEXT_PUBLIC_DOCS_SUPABASE_ANON_KEY`**; сервисный ключ нового проекта — **`DOCS_SUPABASE_SERVICE_ROLE_KEY`** (только сервер, без `NEXT_PUBLIC_`); ключи ИИ — Edge Function / API route. Значения держать в `.env.local` (в репозитории игнорируется маской `.env*`).

Порядок применения миграций и проверка Realtime: [supabase-docs-project-setup.md](./supabase-docs-project-setup.md).

## 8. Выбор модели и ориентировочная стоимость запроса

**UI:** перед отправкой сообщения в ИИ — **выпадающий список (dropdown) модели**. Список короткий: **две модели OpenAI (ChatGPT)** и **две Anthropic (Claude)** — без длинного каталога.

Пример состава v1 (идентификаторы уточнять под актуальные API при внедрении):

| Провайдер | Модель в UI | API id (пример) | Назначение |
|-----------|-------------|-----------------|------------|
| OpenAI | GPT-4o | `gpt-4o` | Основная, качество |
| OpenAI | GPT-4o mini | `gpt-4o-mini` | Дешевле / быстрее |
| Anthropic | Claude Sonnet | актуальный id **Sonnet** из документации API | Основная, качество |
| Anthropic | Claude Haiku | актуальный id **Haiku** из документации API | Дешевле / быстрее |

**Подсчёт стоимости «плюс-минус»:**

- В конфиге приложения для каждой модели хранить **оценочные тарифы**: USD за 1M **input** токенов и за 1M **output** токенов (периодически обновлять вручную или с сайта провайдера).
- Перед отправкой запроса:
  - оценить **input**: системный промпт + история чата + **текущий документ** + текст **прикреплённого к этому сообщению контекста** (другие документы) + новое сообщение пользователя (грубая оценка токенов через приближённые токенайзеры или эвристика «символы / 4» с поправкой);
  - заложить **output**: константа-диапазон (например 256–1024 токенов) или оценка из длины последних ответов.
- Показать пользователю одну строку, например: **«~ $0.03 (±30%)»** или диапазон **«~ $0.02–0.05»**, с подписью что это **оценка**, фактическая сумма после ответа может отличаться (по `usage` из API).
- После ответа ИИ — опционально обновить отображение «фактически ~ $X» по полям `usage.prompt_tokens` / `completion_tokens`, если провайдер их отдаёт.

Так пользователь понимает порядок цены до клика «Отправить», без бухгалтерской точности в UI.

## 10. Открытые решения

- Один чат на документ или отдельный «только ИИ» канал vs смешанный чат людей + ИИ — от этого зависит модель сообщений.
- Формат обмена с ИИ: всегда полный текст документа vs структурированный патч (второй лучше при больших файлах).
- Синхронизация текста документа: **Yjs / CRDT** поверх Supabase Realtime vs упрощённая «последняя версия wins» для очень маленького MVP — trade-off между конфликтами и сложностью.

## 11. Схема БД: от черновика к продакшену

Ниже — что имеет смысл **добавить** к твоим трём таблицам `chats`, `messages`, `documents`.

### Обязательные правки к черновику

- **`documents.chat_id`** — внешний ключ `references chats(id) on delete cascade` (сейчас связь не зафиксирована на уровне БД).
- **Один чат ↔ один документ** (по текущей модели) — `unique (chat_id)` на `documents`, чтобы не повесить два документа на один чат.
- **`messages.role`** — не голый `text`: `check (role in ('user', 'assistant', 'system'))` или Postgres enum. Для OpenAI/Anthropic обычно используют `assistant`, не `ai`, чтобы совпадать с API (можно маппить в UI).
- **`updated_at` на `documents`** — триггер `before update` с `new.updated_at = now()`, иначе поле быстро перестанет быть правдой.

### Сообщения (`messages`) — полезные колонки

| Колонка | Зачем |
|---------|--------|
| `model text null` | Какая модель ответила (для AI-сообщений). |
| `usage jsonb null` | `{ "prompt_tokens", "completion_tokens" }` сырого ответа провайдера — для фактической стоимости и отладки. |
| `metadata jsonb null default '{}'` | Запас: ошибки, latency, id запроса, черновик «показать оценку $»; для user-сообщений — опционально `{ "context_document_ids": [...], "context_excerpt": "..." }` чтобы в UI показать прикреплённый контекст без отдельной таблицы линков. |
| `sort_key bigint` или строго монотонный порядок | Если два сообщения с одинаковым `created_at`, порядок в ленте не плывёт (редко, но дёшево защититься). |

Индекс: **`messages (chat_id, created_at)`** (или `(chat_id, sort_key)`).

### Чаты (`chats`)

- Опционально **`title text`** — заголовок сессии в списке «мои документы».
- Опционально **`default_model text`** — последняя выбранная модель в dropdown для этого чата.

### Контекст из других документов (без таблицы `document_links`)

Отдельная таблица связей **не нужна**: достаточно при отправке сообщения включить выбранный текст других документов в тело промпта (и при желании продублировать в `messages.metadata`, какие `document_id` и под каким углом были приложены — только для отображения в чате и отладки, не как граф ссылок).

### Реалтайм: набор в чате

- **Вариант A:** только **Realtime Broadcast** по `room = chat_id` — без таблицы, статус «печатает» эфемерный.
- **Вариант B:** таблица `chat_typing (chat_id, participant_id text, updated_at)` + периодическая чистка, если нужна аналитика или fallback без broadcast.

Для «простого» MVP чаще хватает A.

### Доступ (Supabase)

- Включить **RLS** на всех таблицах; политики под твою модель доступа (пока общая ссылка — отдельная таблица `chat_invites` или секрет в query, иначе anon откроет всё).
- Индексы по полям, которые пойдут в `where` политик (`auth.uid()`, `chat_id` и т.д.).

### Опционально позже

- **`document_revisions`** — снапшоты `content` до/после ответа ИИ (откат без полноценного CRDT).
- **`content_format text` default 'markdown'`** — если появятся `.txt` vs `.md` с разным рендером.
